# Feature Specification: gI 기반 ToDo 관리 앱

**Feature Branch**: `001-cli-todo-app`  
**Created**: 2026-05-02  
**Status**: Draft  
**Input**: User description: "CLI 기반의 ToDo 관리 앱을 만들어 싶어요. 사용자: 터미널을 사용하는 개인 개발자. 주요 기능: ToDo 항목 추가(제목 필수, 마감일·우선순위 선택), 전체 목록 조회(완료/미완료/우선순위 필터링), 항목 완료 처리(ID로), 항목 삭제(ID로)"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ToDo 항목 추가 (Priority: P1)

개발자가 터미널에서 새로운 할 일 항목을 제목·마감일·우선순위와 함께 등록한다.
등록된 항목에는 고유 ID가 자동 부여되어 이후 완료 처리 및 삭제에 사용된다.

**Why this priority**: 모든 다른 기능(조회, 완료, 삭제)은 항목이 먼저 존재해야 동작한다. 이 기능 없이는 앱 자체가 의미 없으므로 P1이다.

**Independent Test**: `todo add "보고서 작성"` 명령 실행 후 저장소에 항목이 생겼고 고유 ID가 부여됐는지 확인하면 단독 검증이 가능하다. 이것만으로 앱의 핵심 가치(할 일 기록) 전달이 가능하다.

**Acceptance Scenarios**:

1. **Given** 비어 있는 ToDo 목록에서, **When** 사용자가 제목만 입력해 항목을 추가하면, **Then** 고유 ID가 부여된 미완료 항목이 저장되고 터미널에 확인 메시지가 출력된다.
2. **Given** 비어 있는 ToDo 목록에서, **When** 사용자가 제목과 마감일, 우선순위를 함께 입력해 항목을 추가하면, **Then** 세 값이 모두 저장된 항목이 생성된다.
3. **Given** 빈 제목으로 추가를 시도하면, **When** 명령이 실행되면, **Then** 오류 메시지가 출력되고 항목은 저장되지 않는다.
4. **Given** 유효하지 않은 마감일 형식이 입력되면, **When** 명령이 실행되면, **Then** 형식 오류 메시지가 출력되고 항목은 저장되지 않는다.

---

### User Story 2 - 전체 목록 조회 및 필터링 (Priority: P2)

개발자가 터미널에서 저장된 ToDo 항목 전체를 조회하거나, 완료/미완료/우선순위 조건으로 필터링해서 본다.

**Why this priority**: 항목을 추가한 직후 바로 내용을 확인해야 하며, 일상적인 작업 관리의 핵심 흐름이다. P1 다음으로 가장 자주 사용되는 기능이다.

**Independent Test**: 미리 추가된 항목이 있는 상태에서 `todo list` 명령을 실행해 항목이 출력되는지 확인한다. 필터 옵션을 각각 사용해 대상 항목만 출력되는지 확인하면 단독 검증이 완성된다.

**Acceptance Scenarios**:

1. **Given** 항목이 여러 개 있을 때, **When** 필터 없이 목록을 조회하면, **Then** 전체 항목이 ID·제목·마감일·우선순위·완료 여부와 함께 출력된다.
2. **Given** 완료 항목과 미완료 항목이 섞여 있을 때, **When** 미완료 필터로 조회하면, **Then** 미완료 항목만 출력된다.
3. **Given** 완료 항목과 미완료 항목이 섞여 있을 때, **When** 완료 필터로 조회하면, **Then** 완료 항목만 출력된다.
4. **Given** 다양한 우선순위 항목이 있을 때, **When** 특정 우선순위 필터로 조회하면, **Then** 해당 우선순위 항목만 출력된다.
5. **Given** ToDo 목록이 비어 있을 때, **When** 목록을 조회하면, **Then** 빈 목록임을 안내하는 메시지가 출력된다.

---

### User Story 3 - 항목 완료 처리 (Priority: P3)

개발자가 특정 ToDo 항목의 ID를 지정해 완료 상태로 변경한다.

**Why this priority**: 할 일 관리의 핵심 라이프사이클이지만, 조회(P2)로 ID를 먼저 확인해야 사용할 수 있어 P3이다.

**Independent Test**: ID가 알려진 미완료 항목에 대해 `todo done <ID>` 실행 후 해당 항목의 완료 여부가 참으로 바뀌었는지 확인하면 단독 검증이 가능하다.

**Acceptance Scenarios**:

1. **Given** 미완료 항목이 존재할 때, **When** 해당 ID로 완료 처리를 실행하면, **Then** 항목의 완료 상태가 변경되고 확인 메시지가 출력된다.
2. **Given** 이미 완료된 항목에 대해 완료 처리를 다시 실행하면, **When** 명령이 실행되면, **Then** 이미 완료된 항목임을 알리는 메시지가 출력되고 상태는 변하지 않는다.
3. **Given** 존재하지 않는 ID로 완료 처리를 시도하면, **When** 명령이 실행되면, **Then** 해당 ID를 찾을 수 없다는 오류 메시지가 출력된다.

---

### User Story 4 - 항목 삭제 (Priority: P4)

개발자가 특정 ToDo 항목의 ID를 지정해 영구 삭제한다.

**Why this priority**: 유용한 정리 기능이나 추가·조회·완료 처리가 모두 동작해야 의미 있는 흐름이 완성된다. 삭제 없이도 기본 ToDo 관리는 가능하다.

**Independent Test**: ID가 알려진 항목에 대해 `todo delete <ID>` 실행 후 해당 항목이 목록에서 사라졌는지 확인하면 단독 검증이 가능하다.

**Acceptance Scenarios**:

1. **Given** 항목이 존재할 때, **When** 해당 ID로 삭제를 실행하면, **Then** 항목이 저장소에서 제거되고 확인 메시지가 출력된다.
2. **Given** 존재하지 않는 ID로 삭제를 시도하면, **When** 명령이 실행되면, **Then** 해당 ID를 찾을 수 없다는 오류 메시지가 출력된다.
3. **Given** 삭제 후 목록을 조회하면, **When** 목록 명령을 실행하면, **Then** 삭제된 항목이 더 이상 표시되지 않는다.

---

### Edge Cases

- 모든 항목이 삭제된 빈 목록 상태에서 조회하면 빈 목록 안내 메시지가 출력된다.
- 우선순위 없이 추가된 항목의 우선순위 필드는 '없음(none)'으로 표시된다.
- 마감일 없이 추가된 항목의 마감일 필드는 비워서 출력한다.
- 매우 긴 제목(예: 200자 이상)은 항목 추가 시 길이 제한 오류를 반환한다.
- 동일한 항목을 여러 번 완료 처리해도 데이터 손상 없이 안전하게 처리된다.
- 터미널 세션 종료 후 재실행해도 이전 데이터가 유지된다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 제목(필수), 마감일(선택, YYYY-MM-DD 형식), 우선순위(선택: high/medium/low) 를 입력받아 ToDo 항목을 생성해야 한다.
- **FR-002**: 생성된 각 ToDo 항목에는 고유 ID가 자동 부여되어야 한다.
- **FR-003**: 제목이 비어 있으면 항목 생성을 거부하고 오류 메시지를 출력해야 한다.
- **FR-004**: 유효하지 않은 날짜 형식(YYYY-MM-DD 외)이 입력되면 오류 메시지를 출력하고 항목을 저장하지 않아야 한다.
- **FR-005**: 시스템은 저장된 모든 ToDo 항목을 목록으로 출력해야 한다. 각 항목은 ID·제목·마감일·우선순위·완료 여부를 포함한다.
- **FR-006**: 시스템은 완료 여부(완료/미완료) 필터를 지원해야 한다.
- **FR-007**: 시스템은 우선순위(high/medium/low) 필터를 지원해야 한다.
- **FR-008**: 사용자는 항목 ID를 지정해 해당 항목을 완료 상태로 변경할 수 있어야 한다.
- **FR-009**: 이미 완료된 항목에 완료 처리를 중복 실행하면 현재 상태를 안내하고 데이터는 변경하지 않는다.
- **FR-010**: 사용자는 항목 ID를 지정해 해당 항목을 영구 삭제할 수 있어야 한다.
- **FR-011**: 존재하지 않는 ID로 완료 처리 또는 삭제를 시도하면 오류 메시지를 출력해야 한다.
- **FR-012**: ToDo 데이터는 터미널 세션 종료 후에도 로컬에 영속 저장되어야 한다.
- **FR-013**: 모든 명령 실행 결과는 터미널에서 사람이 읽기 쉬운 형태로 출력되어야 한다.

### Constitution Alignment *(mandatory)*

- **CA-001 (CLI Scope)**: 이 기능의 모든 인터페이스는 CLI 명령 형태이며, REST API·GUI·웹 인터페이스는 범위 외다.
- **CA-002 (Test-First)**: 각 User Story의 Acceptance Scenarios가 테스트 코드 작성의 기준이 되며, 구현 전 테스트가 먼저 작성되어야 한다.
- **CA-003 (Layer Separation)**: 도메인 규칙(항목 생성·조회·완료·삭제 로직)은 CLI 입출력 처리 코드와 분리된 레이어에서 구현되어야 한다.
- **CA-004 (Dependency Review)**: 외부 라이브러리 사용 전 표준 라이브러리로 해결 가능한지 검토하고, 사용 시 필요성을 plan에 명시해야 한다.

### Key Entities

- **TodoItem**: 단일 할 일 항목. 속성: ID(고유), 제목(필수, 문자열), 마감일(선택, 날짜), 우선순위(선택: high/medium/low), 완료 여부(불리언), 생성일시.
- **TodoRepository**: 항목 목록의 영속 저장소. 추가·조회·완료 처리·삭제 연산을 담당하며, 저장 매체(파일 등)를 추상화한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 ToDo 항목을 추가하는 작업을 명령 1회로 30초 이내에 완료할 수 있다.
- **SC-002**: 항목 목록 조회 명령은 최소 1,000개 항목 기준 3초 이내에 결과를 출력한다.
- **SC-003**: 필터링 조회(완료/미완료/우선순위)가 정확히 대상 항목만 반환하며 누락·중복이 없다.
- **SC-004**: 완료 처리 및 삭제 명령 실행 후 목록 재조회 시 변경 내용이 즉시 반영된다.
- **SC-005**: 터미널 세션을 닫고 다시 열었을 때 이전에 저장된 모든 항목이 그대로 유지된다.
- **SC-006**: 잘못된 입력(빈 제목, 잘못된 날짜 형식, 존재하지 않는 ID)에 대해 100% 오류 메시지가 출력되고 데이터는 변경되지 않는다.
- **SC-007**: 4가지 핵심 명령(추가·조회·완료·삭제) 각각에 대해 명세된 Acceptance Scenarios를 모두 통과하는 자동화 테스트가 존재한다.

## Assumptions

- 사용자는 단일 로컬 머신에서 혼자 사용하는 개인용 도구이므로 다중 사용자 동시 접근은 고려하지 않는다.
- 마감일은 YYYY-MM-DD 형식을 표준으로 가정하며, 별도 명세가 없으면 이 형식만 허용한다.
- 우선순위 값은 high / medium / low 세 가지로 고정하며, 그 외 값은 오류로 처리한다.
- 데이터는 사용자의 로컬 파일 시스템에 저장하며, 저장 파일 위치의 기본값은 사용자 홈 디렉터리 또는 실행 디렉터리로 가정한다(plan 단계에서 확정).
- 삭제는 영구 삭제이며 복구 기능은 v1 범위 밖이다.
- CLI 도구의 출력은 단색 텍스트를 기본으로 하며, 색상/스타일 지원은 선택 사항이다.
- 항목 제목의 최대 길이는 200자로 가정한다.
